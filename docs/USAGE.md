# Using ringwood — a walkthrough with real output

This doc shows end-to-end conversation flows and the CLI output you'll
actually see. Every block below is copy-pasted from a real session (edits
only for brevity and to swap session-specific ids for placeholders).

## 1. Your first week

### Day 1 — a teammate asks about your team's deploy convention

```
you > our DB migrations are always two-phase: schema first (both versions
      accepted), then app switch. Rollback is forward-only — never drop,
      always add a new migration.

claude > Got it.
          🧠 wiki ← ADD decision/db-two-phase-deploy
```

The Stop hook fired, the classifier tagged this as a `decision`, and the
engine wrote the page. Check:

```
$ npx ringwood show decision/db-two-phase-deploy
```

### Day 3 — the same teammate (or a new one) asks without context

```
claude > 위키에 따르면 두 단계 배포 방식을 씁니다:
           1. 스키마만 배포 — 구/신 양쪽을 허용
           2. 애플리케이션 스위치
         롤백은 forward-only 정책입니다.
```

The citation footer is produced by the `search_wiki` MCP tool — Claude
picked it up without prompting.

```
          🧠 wiki ← UPDATE decision/db-two-phase-deploy
```

Check the diff:


The body now contains both the original policy and the new qualifier; the
summary was regenerated by Sonnet to mention both.

### Day 7 — contradiction

```
you > forget what I said about forward-only. We're switching to full
      rollback with DB snapshots for any migration.

claude > 이전 정책을 무효화하고 새 정책으로 기록합니다.
          🧠 wiki ← SUPERSEDE decision/db-two-phase-deploy →
                   decision/db-full-rollback-with-snapshots
```

Now `timeline` shows what happened:

```
$ npx ringwood timeline --tail 5
- [2026-04-19T02:15:00+00:00] ADD decision/db-two-phase-deploy
- [2026-04-21T10:07:12+00:00] UPDATE decision/db-two-phase-deploy | refinement
```

The old page is **not deleted** — it's stamped `invalid_at: 2026-04-23`
and hidden from search. You can still see it with `--include-invalid`
(planned) or by reading the file.

## 2. Weekly inspection

```
$ npx ringwood stats

📊 Wiki stats (week)
questions answered     14
avg citations / Q      2.3
new pages              9
updated pages          3
invalidated            1

🔗 Top cited
     8×  decision/db-full-rollback-with-snapshots
     5×  concept/prompt-caching-economics
     3×  decision/asia-seoul-timezone-alerts
```

Two signals worth watching:

- **avg citations / Q** — trending up means Claude is finding more relevant
  prior knowledge per question. Trending down means either your questions
  drifted topics, or retrieval needs work.
- **invalidated** — contradictions caught. Zero is suspicious if you're
  iterating actively; >5/week means the schema is volatile and you might
  want to bump `volatility: project` on those pages.

## 3. Integrity

```
$ npx ringwood lint
· orphans (2):
  concept/prompt-caching-economics
  decision/asia-seoul-timezone-alerts
· stale (1):
  entity/pm-alpha
✕ invalidated (3)
```

Orphans are pages nothing links to. They're not broken — they're
candidates for cross-linking. `entity/pm-alpha` is stale (90+ days old
for `volatility: project` pages), so the engine will probably ask to
re-confirm it on next touch.

## 4. Command reference

| Command | Purpose |
|---|---|
| `npx ringwood init` | Register MCP server in `~/.claude.json` |
| `npx ringwood init --with-hook` | Also install the Stop hook |
| `npx ringwood path` | Print the wiki root |
| `npx ringwood doctor` | Health check |
| `npx ringwood stats [--period week\|day\|month] [--json]` | Growth summary |
| `npx ringwood timeline [--tail N]` | Audit log |
| `npx ringwood diff --days N` | Changes since N days ago |
| `npx ringwood list [--kind concept]` | List page ids |
| `npx ringwood show <page_id>` | Print full page markdown |
| `npx ringwood lint` | Broken links, stale, orphans |
| `npx ringwood serve [args…]` | Run the MCP server directly |

## 5. Common questions

### Q: How do I share a wiki with my team?

Today: `git init ~/ringwood && git commit -am 'seed'` and push. Each
teammate runs `npx ringwood init --root <their-clone>` to point at it.
Merge conflicts are rare — pages are single-file — but when they happen
they're normal markdown conflicts.

Planned (Phase 3): HTTP MCP + OAuth so a team can share one server.

### Q: Will this use all my Anthropic credits?

A typical week of heavy use (≈50 Q&As) costs $0.05–0.25 depending on how
many trigger the full decision pipeline. The classifier runs first on
every turn and short-circuits 70–90% of trivia. See `docs/QUALITY_REPORT_PHASE1.md`
for measured numbers.

### Q: Can I edit pages by hand?

Yes. Everything is markdown. Open `~/ringwood/wiki/decision/foo.md` in
any editor and save. Next search call reindexes automatically. Keep the
frontmatter intact or the page won't load — `npx ringwood lint` catches
malformed frontmatter before it hides a page.

### Q: What happens without an API key?

The engine falls back to a rule-based classifier (regex for "remember",
"actually", "we decided", etc.) and a deterministic decision heuristic.
This is the Phase 0 path. It still works and is useful for capturing
explicit saves — but it doesn't catch synthesis, comparisons, or nuanced
contradictions the way the LLM path does. See the
[A/B quality report](./ANSWER_QUALITY_REPORT.md) for what you're leaving
on the table.

### Q: Does it work in English too?

Yes. The trigram tokenizer handles English by the same 3-char window,
and all prompts are language-agnostic. The golden test suite is Korean
because that was our initial tough case.
